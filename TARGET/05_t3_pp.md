# T2. PP 异步化

把 1F1B 从“逻辑调度”做成“真实异步 P2P”。

**建议窗口：D10-D13**

| 子维度       | 任务                                                                                 |
| --------- | ---------------------------------------------------------------------------------- |
| Impl      | 重构 PP send/recv helper：`comm_stream + event`，去掉不必要的 `wait()+synchronize()`；统一 microbatch 状态机 |
| Comm/NCCL | 把每次 `isend/irecv`、wait 点、tensor shape、对端 rank 记录下来；形成 P2P 时序表                      |
| Overlap   | warmup/steady/cooldown 都要 trace；要求 steady-state 里 compute 与 P2P 并存，不是 blocking 伪异步 |
| Trace     | 导出 stage 级 timeline，回答 bubble 在哪、等待来自哪条边                                           |
| Recovery  | PP 模式 save/load 回归；保证不同 stage 恢复后调度状态一致                                            |
| Source    | 读：Megatron PP schedule 入口、你自己 `pipeline_parallel.py` / `pp_communications.py`      |
| Failure   | 复盘 1 个 PP 类问题：`wait()+synchronize()` 抹掉 overlap、activation 生命周期错误、microbatch 次序错   |
| 交付        | `reports/t2_pp_async.md`、P2P 时序图、bubble 分析图                                        |
| 通过标准      | 你能解释“为什么原来是 blocking 1F1B、改完后哪段 latency 被隐藏、还有哪些 bubble 是算法决定的”                    |
